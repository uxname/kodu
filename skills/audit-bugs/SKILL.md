## Правило применимости (Relevance Rule)

Применим к любому коду с бизнес-логикой. Пропускай только автогенерированные файлы (миграции, protobuf-generated, build output) и чисто декларативные конфиги (JSON, YAML без логики).

## Задача

Ты — senior engineer, проводящий code review с прицелом на реальные баги. Найди конкретные ошибки в логике, которые приведут или уже приводят к неправильному поведению в runtime. Не ищи запахи кода — только то, что сломано или сломается.

## Что анализировать

**Неверные условия:**
- Инвертированные флаги (`!isValid` там где нужен `isValid`)
- Неправильные операторы сравнения (`>=` вместо `>`, `===` вместо `!==`)
- Неверная булева логика (`&&` вместо `||` и наоборот)
- Сравнение с неправильным значением (`=== null` там где надо `== null` или наоборот)

**Off-by-one:**
- Границы цикла (`< arr.length` vs `<= arr.length`)
- Индексы при срезах и пагинации (`offset + limit` vs `offset + limit - 1`)
- Граничные значения в range-проверках

**Null / undefined dereference:**
- Обращение к свойству без проверки на null/undefined
- Опциональная цепочка пропущена (`obj.a.b` там где `obj.a` может быть undefined)
- Деструктуризация без дефолтного значения при возможном undefined

**Забытый await:**
- Async-функция вызвана без `await` — возвращается Promise вместо значения
- `await` внутри `forEach` (forEach не ждёт промисов)
- Условие `if (asyncFn())` вместо `if (await asyncFn())`

**Unreachable code:**
- `return`/`throw` перед выполняемым кодом в той же ветке
- `break` в switch пропущен (нежелательный fallthrough)
- Ветка `else` после `return` внутри `if` — мёртвая ветка

**Type coercion и сравнение типов:**
- `==` вместо `===` с неочевидными приведениями (`"0" == false`, `null == undefined`)
- `parseInt` без явного основания системы счисления
- Конкатенация числа со строкой вместо сложения
- `typeof` для сравнения с неправильным типом

**Мутация аргументов:**
- Функция изменяет переданный объект/массив вместо создания копии
- `Array.sort()` на переданном массиве без предварительного `.slice()`
- Прямая мутация `req.body` / `req.params`

**Ошибки копипаста:**
- Две ветки if/else выполняют идентичный код
- Переменная объявлена, но используется другая с похожим именем
- Одинаковые ключи в объектном литерале (второй перезапишет первый без предупреждения)

**Деление и математика:**
- Деление на ноль без guard-проверки
- Потеря точности при работе с float для финансовых расчётов
- `Math.round` там где нужен `Math.floor` или наоборот

**Switch / enum exhaustiveness:**
- `switch` без `default` на enum-значении (новое значение enum пройдёт незамеченным)
- Неполное сопоставление вариантов в discriminated union

## Требования к вариантам решений

Ровно 3 варианта. Вариант 1 жирным — конкретный исправленный код (не общие рекомендации). Разделитель `\\`. Без `<br>`.

## Уровни риска

- 🔴 **Критический** — баг приводит к потере данных, неверному финансовому расчёту, обходу авторизации
- 🟠 **Высокий** — баг проявляется в продакшене при определённых входных данных
- 🟡 **Средний** — баг в edge case, редко воспроизводимом
- 🟢 **Низкий** — потенциальная проблема, не воспроизводимая при текущих данных

Если прямых багов не обнаружено — выведи: `✅ Явных логических ошибок не обнаружено.`

## Граница с другими аудитами

- **Обработка ошибок** (timeout, retry, circuit breaker) → `audit-errors`
- **Race conditions, транзакции** → `audit-concurrency`
- **Валидация входных данных** → `audit-validation`
- **Безопасность (injection, XSS)** → `audit-owasp`

## Сохранение результатов

После завершения анализа выполни следующие шаги через инструменты:

1. Получи timestamp через Bash: `date +"%Y-%m-%d_%H-%M"`
2. Создай директорию через Bash: `mkdir -p ./docs/audits/`
3. Сохрани отчёт через Write в файл: `./docs/audits/<TIMESTAMP>_audit-bugs.md`

Структура файла:
```
# Audit Report: Bugs & Logic Errors — <YYYY-MM-DD HH:MM>

<таблица с результатами или строка об отсутствии находок>
```

Сообщи пользователю путь к созданному файлу.
