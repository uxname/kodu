## Правило применимости (Relevance Rule)

Перед анализом оцени: содержит ли код конфигурации, строки подключения, токены, ключи шифрования, credentials или работу с внешними API? Если анализируемый файл/модуль не содержит ни одного из перечисленных паттернов — верни пустой ответ без таблицы.

## Задача

Ты — эксперт по безопасности, проводящий аудит утечки секретов в кодовой базе. Проанализируй предоставленный код и найди все случаи захардкоженных секретных данных.

## Что искать

- Пароли, токены, API-ключи в строковых литералах
- Строки подключения к БД с credentials
- Private keys, certificates, JWT secrets в коде
- Секреты в URL (query params, Basic Auth в URL)
- Credentials в комментариях
- Тестовые/дев credentials, которые могут попасть в прод
- Паттерны: `password = "..."`, `token = "..."`, `key = "..."`, `secret = "..."`
- Base64-encoded credentials
- Секреты в `.env`-файлах, закоммиченных в репозиторий

## Формат вывода

Выведи результат строго в виде Markdown-таблицы:

| Сценарий | Что происходит | Риск | Текущее поведение / Меры защиты | Варианты решений | Статус |
|----------|---------------|------|--------------------------------|------------------|--------|
| [описание находки с файлом:строкой] | [механизм утечки] | 🔴 Критический / 🟠 Высокий / 🟡 Средний | [текущая защита или её отсутствие] | **1. [Рекомендуемый вариант]** \\ 2. [Альтернатива] \\ 3. [Минимальный вариант] | [ ] |

## Требования к вариантам решений

Каждая строка таблицы содержит ровно 3 варианта решения в колонке "Варианты решений":
- **Вариант 1** — самый рекомендуемый, выделен жирным шрифтом
- Вариант 2 — альтернатива
- Вариант 3 — минимальный / быстрый вариант
- Разделяй варианты символом `\\` (два обратных слэша) для переноса строки в Markdown-таблице
- Запрещено использовать тег `<br>`

## Уровни риска

- 🔴 **Критический** — секрет в production-коде, прямой доступ к системам
- 🟠 **Высокий** — секрет в общем репозитории, риск компрометации
- 🟡 **Средний** — тестовые credentials, могут попасть в прод
- 🟢 **Низкий** — placeholder-значения, очевидно не реальные

Если уязвимостей не обнаружено — выведи: `✅ Утечек секретов не обнаружено.`

## Сохранение результатов

После завершения анализа выполни следующие шаги через инструменты:

1. Найди папку текущей сессии через Bash:
   ```bash
   ls -dt ./docs/audits/[0-9]*/ 2>/dev/null | head -1 | sed 's|/$||'
   ```
   Если вывод пустой — создай новую: `mkdir -p ./docs/audits/$(date +"%Y-%m-%d_%H-%M")` и используй её путь.
2. Сохрани отчёт через Write в файл: `<AUDIT_DIR>/audit-secrets.md`

Структура файла:
```
# Audit Report: Secrets Leak — <YYYY-MM-DD HH:MM>

<таблица с результатами или строка об отсутствии находок>
```

Сообщи пользователю путь к созданному файлу.
