## Задача

Ты — системный архитектор. Смотришь на систему сверху: не на файлы и функции, а на крупные части (Frontend, Backend, Database, очередь сообщений, кэш, Docker, внешние сервисы и т.п.). Задача — найти все такие части, понять как они связаны, и описать что произойдёт в худших сценариях.

## Шаг 2 — Обнаружение компонентов системы

Смотри на инфраструктуру и конфигурацию — не на исходный код:

```bash
# Docker и инфраструктура
cat docker-compose.yml 2>/dev/null || cat docker-compose.yaml 2>/dev/null
ls Dockerfile* 2>/dev/null
ls kubernetes/ k8s/ helm/ infra/ deploy/ 2>/dev/null

# Базы данных и внешние сервисы
grep -rh "postgres\|mysql\|mongo\|redis\|rabbitmq\|kafka\|elasticsearch\|s3\|smtp\|sendgrid\|stripe\|twilio" \
  docker-compose.yml docker-compose.yaml .env.example .env 2>/dev/null | head -20

# Тип проекта (монолит, монорепо, микросервисы)
ls package.json packages/ apps/ services/ 2>/dev/null
cat package.json 2>/dev/null | grep -E '"name"|"scripts"' | head -10

# Конфигурация окружений
ls .env* 2>/dev/null
cat .env.example 2>/dev/null || cat .env.sample 2>/dev/null | head -40

# CI/CD
ls .github/workflows/ .gitlab-ci.yml .circleci/ 2>/dev/null
```

На основе этих данных **самостоятельно определи архитектурные компоненты системы** — крупные части типа:

> Frontend, Backend API, База данных, Redis/кэш, Очередь сообщений, Worker/фоновые задачи, CDN, Внешние API (платёжки, почта, SMS), Docker/инфраструктура, CI/CD, ...

Не ограничивайся этим списком — называй компоненты по тому, что реально нашёл в проекте.

Для каждого компонента:
- Имя
- Что делает (1 строка)
- Как запускается / откуда берётся (Docker-сервис, облако, внешний API, ...)

## Шаг 3 — Карта взаимодействий

Построй граф: кто с кем общается.

```bash
# Подтверди связи
grep -rh "http\|grpc\|amqp\|redis\|postgres\|DATABASE_URL\|REDIS_URL\|API_URL" \
  .env.example docker-compose.yml 2>/dev/null | head -30
```

Граф в текстовом виде:
```
Frontend → Backend API → Database
Backend API → Redis (кэш)
Backend API → Worker (очередь)
Worker → External Email API
```

Для каждой связи отметь:
- `✅` — есть retry / fallback / circuit breaker
- `⚠` — частичная защита
- `❌` — нет защиты, сбой одной стороны = сбой другой

## Шаг 4 — Анализ сценариев сбоя

### 4.1 — Худшие сценарии для каждого компонента

Для каждого компонента задай вопрос: **что может пойти не так?**

- Компонент упал / недоступен — что видит пользователь?
- Компонент тормозит — накапливается ли очередь / timeout-шторм?
- Данные повреждены или потеряны — как система об этом узнаёт?
- Компонент перезапустился — теряется ли состояние?
- Кончился ресурс (память, диск, соединения) — что происходит?
- Обновление/миграция прошла с ошибкой — откат возможен?

Подтверди каждый сценарий конкретным файлом или конфигурацией (`docker-compose.yml:line`, `.env.example`, `Dockerfile`, и т.п.).

### 4.2 — Каскадные сбои

Возьми все связи с `❌` из Шага 3. Для каждой:
1. Компонент B упал → что происходит с A?
2. A передаёт сбой дальше → кто ещё ломается?
3. Прослеживай цепочку до конечного пользователя или потери данных

## Severity

| Уровень | Критерий |
|---------|----------|
| 🔴 Критический | Потеря данных, нарушение безопасности, полный отказ сервиса без восстановления |
| 🟠 Высокий | Деградация без полного отказа, проблемы под нагрузкой |
| 🟡 Средний | Некорректное поведение в редких случаях |
| 🟢 Низкий | Технический долг, отсутствие мониторинга в некритичных местах |

## Правила верификации

1. **Уровень системы** — компоненты это Frontend, Backend, DB, Docker и т.п., не классы и функции
2. **Только реальные конфиги** — каждый сценарий подкреплён файлом (`docker-compose.yml`, `.env.example`, `Dockerfile`, и т.п.)
3. **Нет спекуляций** — если не нашёл доказательства, пиши `⚠ не проверено — {причина}`
4. **Межкомпонентные** — минимум 3 каскадных сценария, иначе секцию не добавляй
5. **Критические задачи** — только 🔴/🟠. 🟡/🟢 не включай
6. **Варианты решений** — только для сценариев без ✅
7. **Исправлено** — `Нет` по умолчанию

## Language Rule

Результаты аудита должны быть написаны простым и понятным языком. Избегай сложных терминов, жаргона и абстрактных понятий без необходимости. Общепринятые технические термины (Docker, HTTP, API, JSON, URL) допустимы. Описывай проблемы так, чтобы они были понятны разработчику любого уровня, а не только узкому специалисту в данной области.
