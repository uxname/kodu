## Задача

Ты — архитектор надёжности. Выяви всех "действующих лиц" системы (компоненты) и найди худшие сценарии: что может пойти не так в каждом из них и при взаимодействии между ними.

LATEST=$(ls -dt ./docs/audits/[0-9]*/ 2>/dev/null | head -1)
SESSION=${LATEST:-./docs/audits/$(date +"%Y-%m-%d_%H-%M")}

## Шаг 2 — Обнаружение компонентов

Собери общую картину системы:

```bash
# Структура проекта
find . -maxdepth 3 -type d | grep -v node_modules | grep -v .git | grep -v dist | grep -v coverage

# Точки входа и конфигурация
ls ./src/*.ts 2>/dev/null; cat package.json | grep -E '"main"|"scripts"' | head -10

# Внешние зависимости
cat package.json | grep -A100 '"dependencies"' | head -40
```

На основе этих данных **самостоятельно определи логические компоненты** — "действующих лиц" системы. Не привязывайся к жёстким категориям: компонент — это любая часть системы с чёткой ответственностью (сервис, слой, внешняя система, инфраструктурный элемент). Количество компонентов не ограничено.

Для каждого компонента:
- Дай ему имя
- Опиши что он делает (1 строка)
- Укажи ключевые файлы

---

## Шаг 3 — Карта взаимодействий

Для каждого компонента определи: кто его вызывает и от кого он сам зависит.

```bash
# Подтверди связи кодом
grep -rn "import.*from\|require(" ./src --include="*.ts" --include="*.js" 2>/dev/null \
  | grep -v node_modules | grep -v "\.test\.\|\.spec\." \
  | grep "{ключевое_слово_компонента}" | head -20
```

Построй текстовый граф на компонентном уровне:
```
ComponentA → ComponentB → ComponentC
ComponentA → ComponentD
```

Для каждой связи отметь:
- `✅` — есть обработка ошибки на границе
- `⚠` — частичная обработка
- `❌` — нет обработки, сбой одного = сбой другого

## Шаг 4 — Анализ сценариев сбоя

### 4.1 — Худшие сценарии для каждого компонента

Для каждого компонента задай вопрос: **что может пойти не так?** Рассмотри:
- Что если компонент недоступен или упал?
- Что если он вернул неожиданный результат (null, пустой список, неверный формат)?
- Что если у него кончился ресурс (память, соединения, дисковое место)?
- Что если внешняя зависимость, которую он использует, перестала работать?
- Есть ли состояния гонки если его вызвать параллельно?

Опирайся на реальный код — каждый сценарий подтверди `file:line`.

### 4.2 — Каскадные сбои (межкомпонентные)

Возьми все связи с `❌` из Шага 3. Для каждой:
1. Компонент B упал → что происходит с A?
2. A передаёт сбой дальше → кто ещё ломается?
3. Прослеживай цепочку до внешней границы (ответ пользователю / данные в БД)

## Severity

| Уровень | Критерий |
|---------|----------|
| 🔴 Критический | Потеря данных, нарушение безопасности, полный отказ сервиса без восстановления |
| 🟠 Высокий | Деградация без полного отказа, гонки под нагрузкой |
| 🟡 Средний | Некорректное поведение в edge-cases, незаметно неверные данные |
| 🟢 Низкий | Технический долг, отсутствие логирования в некритичных путях |

## Правила верификации

1. **Только реальный код** — каждый сценарий подкреплён `file:line` в "Текущее поведение"
2. **Нет спекуляций** — если не нашёл доказательства, пиши `⚠ код не проверен — {причина}`
3. **Межкомпонентные** — минимум 3 каскадных сценария, иначе секцию не добавляй
4. **Критические задачи** — только 🔴/🟠. 🟡/🟢 не включай
5. **Варианты решений** — только для сценариев без ✅
6. **Исправлено** — `Нет` по умолчанию
7. **Без повторов с /audit** — матрица = системный уровень, не дублируй code-level находки из аудита

## Language Rule

Результаты аудита должны быть написаны простым и понятным языком. Избегай сложных терминов, жаргона и абстрактных понятий без необходимости. Общепринятые технические термины (Docker, HTTP, API, JSON, URL) допустимы. Описывай проблемы так, чтобы они были понятны разработчику любого уровня, а не только узкому специалисту в данной области.
