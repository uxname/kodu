## Задача

Ты — архитектор безопасности и надёжности. Построй матрицу отказов для системы: обнаружь компоненты, проанализируй как каждый может упасть, что при этом сломается в других компонентах, и предложи решения.

Инспирация: документ `03-failure-matrix.md` в san-sdk — компоненты → сценарии → риски → решения.

## Шаг 2 — Обнаружение компонентов (авто)

Выполни все команды, собери данные:

```bash
# 1. Структура src/
find ./src -maxdepth 3 -type d 2>/dev/null | head -30

# 2. Ключевые файлы по ролям
grep -rl "Router\|createRouter\|express()\|app\.use" ./src 2>/dev/null | head -15
grep -rl "export class.*Service\|export.*Service\b" ./src 2>/dev/null | head -15
grep -rl "export class.*Repository\|export class.*Store\|export class.*Cache" ./src 2>/dev/null | head -10
grep -rl "middleware\|Middleware" ./src 2>/dev/null | head -10
grep -rl "auth\|Auth\|jwt\|JWT\|apiKey\|API_KEY" ./src 2>/dev/null | head -10

# 3. Граф импортов (кто кого вызывает)
grep -rh "^import.*from\|require(" ./src --include="*.ts" --include="*.js" 2>/dev/null | \
  grep -v node_modules | grep -v "\.test\.\|\.spec\." | sort | uniq -c | sort -rn | head -30

# 4. Entry points и config
ls ./src/*.ts 2>/dev/null || ls ./src/index.* 2>/dev/null
cat ./src/app.ts 2>/dev/null || cat ./src/main.ts 2>/dev/null || cat ./src/index.ts 2>/dev/null | head -60
```

На основе данных определи компоненты. Типичные группы:
- **Entry/Bootstrap** — `index.ts`, `main.ts`, `app.ts`
- **API Layer** — routes, controllers, handlers
- **Auth** — middleware/auth, jwt, apiKey
- **Middleware** — errorHandler, requestId, rateLimiter
- **Business Logic** — services
- **Data Layer** — repositories, stores, data.ts
- **Config/Utils** — config.ts, types.ts, errors.ts
- **External** — HTTP clients, message queues, caches
- **Deployment** — Dockerfile, docker-compose, CI

## Шаг 3 — Граф зависимостей (доказательный)

### 3.1 — Собери рёбра из реального кода

Для каждого компонента из Шага 2 выполни:

```bash
# Найти все импорты из каждого файла компонента
grep -n "^import\|require(" {файл_компонента} 2>/dev/null | grep -v node_modules
```

Для каждого `import X from './path'` — это ребро `текущий_компонент → X`.

Построй таблицу рёбер с доказательствами:

| Caller | Called | Файл и строка | Тип вызова |
|--------|--------|---------------|------------|
| Routes | AuthMiddleware | `routes/users.ts:3` | middleware |
| Routes | UserService | `routes/users.ts:5` | service call |
| UserService | UserRepository | `services/user.ts:1` | data access |
| App | ErrorHandler | `app.ts:18` | middleware |

### 3.2 — Для каждого ребра: что сломается если Called упадёт?

Пройди по каждой строке таблицы. Для ребра `A → B` прочитай код вызова в A и ответь:

- Есть ли `try/catch` вокруг вызова B?
- Есть ли fallback если B вернул `null`/`undefined`/ошибку?
- Что получит пользователь если B недоступен?

Отметь каждое ребро:
- `✅` — есть обработка ошибки на границе
- `⚠` — частичная обработка (логирует но не восстанавливается)
- `❌` — нет обработки, ошибка B = краш A

### 3.3 — Итоговый граф + аномалии

Выведи текстовый граф:
```
ComponentA → ComponentB → ComponentC
ComponentA → ComponentD
```

Отметь отдельно:
- **Незащищённые рёбра** (`❌`): `A → B` без обработки — главные кандидаты для межкомпонентных сценариев
- **Fan-out > 3**: компонент вызывает 4+ других — god-object, единая точка отказа
- **Циклы**: `A → B → A` — взаимная блокировка при сбое

## Шаг 4 — Анализ сценариев сбоя

### 4.1 — Per-component сценарии

Для каждого компонента: прочитай его код, затем генерируй сценарии на основе того что реально увидел — не угадывай, опирайся на код.

### 4.2 — Межкомпонентные каскады из незащищённых рёбер

Возьми все рёбра `❌` из Шага 3.2. Каждое незащищённое ребро `A → B` — потенциальный каскад. Для каждого:

1. Прочитай код вызова в A (`файл:строка` из таблицы рёбер)
2. Определи: что вернёт A своему вызывающему если B упал?
3. Найди вызывающего A в таблице рёбер — есть ли там защита?
4. Прослеживай цепочку пока не найдёшь защиту или внешнюю границу

Это и есть каскадный сценарий для секции "Межкомпонентные сценарии".

## Severity для сценариев

| Уровень | Критерий |
|---------|----------|
| 🔴 Критический | Потеря данных, security breach, полный отказ сервиса без восстановления |
| 🟠 Высокий | Деградация без полного отказа, race condition под нагрузкой, open position без stop-loss |
| 🟡 Средний | Некорректное поведение в edge-cases, silently wrong data |
| 🟢 Низкий | Технический долг, отсутствие логирования в некритичных путях |

## Правила верификации

1. **Только реальный код** — каждый сценарий подкреплён `file:line` в "Текущее поведение"
2. **Нет спекуляций** — если не нашёл доказательства, пиши `⚠ код не проверен — {причина}`
3. **Межкомпонентные** — минимум 3 каскадных сценария, иначе секцию не добавляй
4. **Критические задачи** — только 🔴/🟠 без ✅. 🟡/🟢 не включай
5. **Варианты решений** — только для сценариев без ✅. Зачёркнутые сценарии — без блока решений
6. **Без повторов с /audit** — не дублируй check-level (LOG-*, BUG-*, etc.) находки из аудита. Матрица = системный уровень, а не code-level
7. **Исправлено** — `Нет` по умолчанию. Зачёркнутые строки (~~текст~~) → `✅ Да`

## Language Rule

Результаты аудита должны быть написаны простым и понятным языком. Избегай сложных терминов, жаргона и абстрактных понятий без необходимости. Общепринятые технические термины (Docker, HTTP, API, JSON, URL) допустимы. Описывай проблемы так, чтобы они были понятны разработчику любого уровня, а не только узкому специалисту в данной области.
