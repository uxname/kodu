## Задача

Ты — системный архитектор-исследователь. Твоя задача — самостоятельно изучить проект,
определить его фактическую архитектуру и составить карту рисков (матрицу сбоев).

**Принцип работы:** Не полагайся на жесткие grep-шаблоны и предопределенные списки технологий.
Используй bash-команды только для первичной навигации. Основной анализ проводи путем
осмысленного чтения файлов конфигурации, манифестов зависимостей и кода инициализации клиентов.
Адаптируйся к любому стеку — даже если проект использует редкие или нестандартные технологии.

## Шаг 2 — Динамическое обнаружение компонентов (Exploratory Analysis)

Не делай предположений о стеке заранее. Действуй как исследователь:

### 2.1. Быстрая навигация по структуре

Используй базовые команды для первичного ознакомления с топологией проекта.
Эти команды — **рекомендательные инструменты**, а не жесткий сценарий:

```bash
# Общая структура верхнего уровня
ls -la

# Контейнеризация и деплой
ls -la docker-compose* Dockerfile* 2>/dev/null; ls kubernetes/ k8s/ helm/ deploy/ infra/ 2>/dev/null

# Переменные окружения и конфиги
ls .env* config/ 2>/dev/null; ls *.example 2>/dev/null

# Манифесты зависимостей
ls package.json requirements.txt Gemfile go.mod Cargo.toml pom.xml build.gradle 2>/dev/null

# CI/CD
ls .github/workflows/ .gitlab-ci.yml .circleci/ 2>/dev/null
```

### 2.2. Глубокий анализ (Deep Scan)

Если файлы обнаружены — **открой и изучи их содержимое**, а не просто фиксируй наличие.
Прочитай ключевые файлы конфигурации полностью:

- **Файлы управления пакетами** — проанализируй секции `dependencies`/`require`/`devDependencies`.
  Выяви используемые драйверы БД, клиенты очередей, SDK внешних сервисов, интеграционные библиотеки.
  *Пример: `package.json` содержит `pg`, `redis`, `@stripe/stripe-js` → проект использует PostgreSQL, Redis, Stripe.*

- **Файлы переменных окружения** (`.env`, `.env.example`) — прочитай полностью.
  Определи все внешние зависимости: URL подключений, API-ключи, хосты, порты.

- **Конфигурационные файлы фреймворков** — ищи и читай `application.yml`, `settings.py`,
  `config.js`, `next.config.js`, `nuxt.config.ts`, `prisma/schema.prisma`, `typeorm.config.ts`,
  `drizzle.config.ts`, `knexfile.js` и т.д.

### 2.3. Определение архитектурной парадигмы

Проанализируй структуру папок, чтобы определить тип архитектуры:

- **Монолит:** единый `package.json`/`go.mod` в корне, весь код в `src/` или `app/`
- **Микросервисы:** множество папок `services/`, `apps/`, `packages/` с собственными манифестами
- **Serverless:** наличие `serverless.yml`, `functions/`, `lambda/`, `workers/`
- **Монорепо:** наличие `lerna.json`, `turbo.json`, `nx.json`, `workspaces` в `package.json`

### 2.4. Формирование списка компонентов

На основе собранных данных составь список **фактических компонентов системы**:

| Пример | Размещение | Как обнаружено |
|--------|------------|----------------|
| PostgreSQL | Docker-контейнер | `docker-compose.yml` + `pg` в `package.json` |
| Stripe Payments | Внешний API | `@stripe/stripe-js` + `STRIPE_KEY` в `.env` |
| Auth Service | Микросервис | Папка `services/auth/` с `Dockerfile` |
| Redis Cache | Docker-контейнер | `redis` в зависимостях + `REDIS_URL` в `.env` |

Для каждого компонента укажи:
- **Название**
- **Роль в системе** (1 строка)
- **Тип размещения** (Docker-контейнер, внешнее облако, SaaS, встроенная библиотека и т.д.)
- **Как обнаружен** (ссылка на конкретный файл и строку)

## Шаг 3 — Карта взаимодействий и протоколов (Lifecycle Tracing)

Не ищи адреса через простой grep — **проследи жизненный цикл конфигурации**.

### 3.1. Поиск точек инициализации соединений

Найди в коде места, где импортируются и инициализируются клиенты:

- Клиенты БД: `new Pool(...)`, `createClient(...)`, `PrismaClient(...)`, `mongoose.connect(...)`
- Брокеры сообщений: `amqp.connect(...)`, `kafka.Consumer(...)`, `bull(...)`, `celery(...)`
- Внешние SDK: `new Stripe(...)`, `new SendGrid(...)`, `new AWS(...)`, `new OpenAI(...)`

**Открой эти файлы и прочитай параметры инициализации:**
- Передаются ли таймауты? (`connectionTimeout`, `requestTimeout`, `timeout`)
- Настроены ли повторные попытки? (`maxRetries`, `retryStrategy`, `reconnect`)
- Указан ли размер пула соединений? (`poolSize`, `maxConnections`, `max`)
- Есть ли `keepAlive`, `heartbeat`, `pingInterval`?

### 3.2. Анализ межсервисных контрактов

Найди формальные контракты взаимодействия:

- `.proto` файлы (gRPC)
- GraphQL схемы (`.graphql`, `schema.graphql`)
- OpenAPI/Swagger спецификации (`openapi.yml`, `swagger.json`)
- Папки `types/`, `contracts/`, `shared/` с общими типами

Определи, какие компоненты общаются по каким протоколам (REST, gRPC, GraphQL, AMQP, WebSocket).

### 3.3. Построение графа

Построй граф: кто с кем общается.

```
Frontend → Backend API (REST/gRPC)
Backend API → Database (SQL)
Backend API → Redis (TCP)
Backend API → Worker (AMQP — очередь)
Worker → External Email API (REST)
```

Для каждой связи укажи уровень защищенности:
- `✅` — есть retry/fallback/circuit breaker/таймауты (подтверждено кодом инициализации)
- `⚠` — базовая защита (только таймауты, без retry)
- `❌` — защиты не обнаружено (клиент инициализируется без параметров отказоустойчивости)

## Шаг 4 — Анализ рисков и каскадных сбоев (Contextual Risk Analysis)

Применяй контекстное архитектурное мышление: сопоставляй библиотеки из Шага 2 с конфигурациями
инициализации из Шага 3. Если код инициализации не использует параметры отказоустойчивости,
которые предоставляет драйвер, — это реальный риск.

### 4.1. Анализ для каждого компонента

Для каждого обнаруженного компонента оцени:

1. **Недоступность** — что видит пользователь? Копятся ли запросы? Есть ли fallback?
2. **Замедление** — есть ли таймауты? Риск истощения пула соединений?
3. **Потеря состояния** — данные в памяти? Есть ли персистентность?
4. **Исчерпание ресурсов** — настроены ли лимиты (память, диск, соединения)?
5. **Ошибка обновления/миграции** — есть ли механизм отката?

**Каждую проблему подтверждай ссылкой на конкретный файл и строку.** Если риск обнаружен
через отсутствие параметров в коде инициализации — укажи это явно.

*Пример:* `src/db.ts:12 — Pool инициализирован без `connectionTimeout`, `maxRetries` и `keepAlive``

### 4.2. Каскадные (межкомпонентные) сценарии

Строй цепочки сбоев на основе **реального пути запроса** в коде, а не по шаблону.
Проследи путь от контроллера до внешнего вызова:

1. Выбери точку входа (контроллер, handler, resolver)
2. Проследи цепочку вызовов через сервисы до БД/внешнего API
3. Определи, что ломается на каждом шаге при отказе следующего звена

**Минимум 3 цепочки.** Формат:

```
# X1: Frontend → Backend → Payments API
Триггер: Payments API недоступен (5xx)
Каскад: Backend запросы зависают → пул соединений исчерпан → новые запросы падают →
  Frontend показывает бесконечный спиннер
Риск: 🔴
Подтверждение: `src/services/payment.ts:15` — Stripe client без таймаута
```

## Правила верификации

1. **Уровень системы** — компоненты это Frontend, Backend, DB, Docker и т.п., не классы и функции
2. **Только реальные конфиги** — каждый сценарий подкреплён файлом и строкой (`src/db.ts:12`)
3. **Анализ отсутствия** — если в коде инициализации клиента нет параметров отказоустойчивости — фиксируй как риск с пометкой `отсутствует параметр`
4. **Нет спекуляций** — если не нашёл доказательства, пиши `⚠ не проверено — {причина}`
5. **Межкомпонентные** — минимум 3 каскадных сценария, иначе секцию не добавляй
6. **Критические задачи** — только 🔴/🟠. 🟡/🟢 не включай
7. **Варианты решений** — только для сценариев без ✅
8. **Исправлено** — `Нет` по умолчанию
