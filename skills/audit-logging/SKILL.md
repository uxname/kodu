## Правило применимости (Relevance Rule)

Перед анализом оцени: содержит ли код вызовы logger, console.log/error/warn, запись в файлы логов, middleware для логирования или аудит-треки? Если анализируемый файл не содержит никакого логирования — верни пустой ответ без таблицы.

## Чеклист

| Check ID | Проверка |
|----------|----------|
| LOG-01 | Нет console.log в production коде |
| LOG-02 | Нет PII (email, phone, name) в log messages |
| LOG-03 | Нет secrets/tokens в logs |
| LOG-04 | Request ID присутствует для трейсинга |
| LOG-05 | Структурированный формат (JSON) в production |
| LOG-06 | Критические операции (auth, create, delete) логируются |
| LOG-07 | Нет log injection (user input санитизируется) |

## Правила верификации

1. **Только чеклист**: оценивай ТОЛЬКО проверки выше. Не добавляй новые.
2. **Нет доказательства = ✅ PASS**: не можешь указать `файл:строка` — ставь PASS.
3. **Baseline приоритетен**: check_id есть в `docs/audit-baseline.yml` → `⏸ ACCEPTED`.
4. **Только 🔴/🟠 FAIL требуют решения**: 🟡/🟢 — решение необязательно.

## Контекст анализа

**Безопасность логов:**
- Логирование паролей, токенов, PII (персональные данные)
- Логирование полных тел запросов/ответов с credentials
- Секреты в stack traces

**Качество логов:**
- Логи без контекста (нет request ID, user ID, trace ID)
- Неправильный уровень (DEBUG в prod, ERROR для info-событий)
- Отсутствие структурированных логов (plain text вместо JSON)
- Дублирующиеся логи в цепочке вызовов

**Избыточность:**
- console.log в production-коде
- Логирование каждой итерации цикла
- Verbose-логи без флага условия

**Полнота:**
- Критические операции без логов (финансовые транзакции, аутентификация)
- Отсутствие логов ошибок при catch-блоках
- Нет audit log для изменения критических данных

## Формат вывода

| Check ID | Проверка | Статус | Доказательство | Решение |
|----------|----------|--------|----------------|---------|
| LOG-01 | Нет console.log в production коде | ✅ PASS | — | — |
| LOG-02 | Нет PII в log messages | ❌ FAIL 🔴 | `auth/login.ts:34` | **1. Удалить email из лога, логировать только userId** \\ 2. Маскировать PII через log sanitizer \\ 3. Заменить на структурированный лог без PII |
| LOG-04 | Request ID присутствует для трейсинга | ⏸ ACCEPTED | — | В baseline: трейсинг через внешний сервис |

Статусы: `✅ PASS` / `❌ FAIL 🔴` / `❌ FAIL 🟠` / `❌ FAIL 🟡` / `❌ FAIL 🟢` / `⏸ ACCEPTED`

Для `❌ FAIL`: ровно 3 варианта решения, разделитель `\\`, вариант 1 жирным.

Если все PASS — выведи: `✅ Логирование соответствует best practices.`
