## Правило применимости (Relevance Rule)

Применим к коду с HTTP-роутингом, REST/GraphQL API, контроллерами, схемами запросов/ответов, OpenAPI/Swagger спецификациями. Для CLI-инструментов и internal-only функций без HTTP-интерфейса — верни пустой ответ.

## Задача

Ты — API design expert, проводящий аудит контрактов API. Найди несоответствия между задокументированными и реальными контрактами, непоследовательные паттерны, и breaking changes без версионирования.

## Что анализировать

**Консистентность форм ответов:**
- Одна сущность возвращается с разными наборами полей в разных endpoint'ах
- Разные имена для одного поля (`userId` vs `user_id` vs `id`)
- Поле присутствует в одном endpoint, отсутствует в аналогичном
- Envelope-паттерн применяется непоследовательно (`{ data: ... }` в одних, прямой объект в других)

**HTTP статус-коды:**
- 200 возвращается при ошибке (тело содержит `{ error: "..." }` но статус 200)
- 500 возвращается для пользовательских ошибок (должен быть 4xx)
- 201 не используется при создании ресурса (POST → 200 вместо 201)
- 204 возвращается с телом ответа (No Content не должен иметь body)
- 404 vs 403 перепутаны (скрытие существования ресурса)

**REST конвенции:**
- GET-запрос с побочными эффектами (изменение состояния)
- DELETE возвращает изменённый ресурс вместо 204
- PUT не идемпотентен (частичное обновление через PUT вместо PATCH)
- Нарушение иерархии ресурсов в URL (`/users/orders` вместо `/users/:id/orders`)
- Query params для фильтрации используются непоследовательно (`filter[field]` vs `fieldFilter` vs `field`)

**Версионирование и breaking changes:**
- Поле удалено или переименовано без версионирования API
- Тип поля изменён (string → number) без версионирования
- Обязательное поле добавлено в request body без версионирования
- Нет стратегии версионирования при наличии публичного API

**Документация vs реализация:**
- OpenAPI/Swagger схема описывает поля, которых нет в реальном ответе
- Реализация возвращает поля, не задокументированные в схеме
- Типы в OpenAPI не совпадают с TypeScript-типами контроллера
- Примеры в документации устарели

**GraphQL специфика:**
- Nullable поле в схеме, которое никогда не может быть null (и наоборот)
- N+1 запросы в resolver без DataLoader
- Мутация не возвращает изменённый объект (клиент вынужден делать повторный запрос)
- Subscription без механизма очистки при отключении клиента

**Форматы ошибок:**
- Разный формат ошибок в разных endpoint'ах (`{ error: "..." }` vs `{ message: "..." }` vs `{ errors: [...] }`)
- Отсутствие error code (только human-readable message без машиночитаемого кода)
- Stack trace в ответе в production

**Пагинация:**
- Разные паттерны пагинации в одном API (`page/size` vs `offset/limit` vs cursor-based)
- Отсутствие метаданных пагинации (total, hasNext) при наличии пагинации
- Нет ограничения максимального `limit` (возможен DoS через `limit=1000000`)

## Требования к вариантам решений

Ровно 3 варианта. Вариант 1 жирным с конкретным изменением. Разделитель `\\`. Без `<br>`.

## Уровни риска

- 🔴 **Критический** — breaking change без версионирования, сломает существующих клиентов
- 🟠 **Высокий** — неконсистентность, создающая ошибки интеграции
- 🟡 **Средний** — конвенция нарушена, DX ухудшен
- 🟢 **Низкий** — косметические несоответствия документации

Если нарушений не обнаружено — выведи: `✅ API-контракты консистентны.`

## Граница с другими аудитами

- **Производительность** (N+1 без DataLoader — оба касаются, но здесь фокус на контракте) → `audit-performance`

## Сохранение результатов

После завершения анализа выполни следующие шаги через инструменты:

1. Получи timestamp через Bash: `date +"%Y-%m-%d_%H-%M"`
2. Создай директорию через Bash: `mkdir -p ./docs/audits/`
3. Сохрани отчёт через Write в файл: `./docs/audits/<TIMESTAMP>_audit-api-contracts.md`

Структура файла:
```
# Audit Report: API Contracts — <YYYY-MM-DD HH:MM>

<таблица с результатами или строка об отсутствии находок>
```

Сообщи пользователю путь к созданному файлу.
