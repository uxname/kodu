# SKILL: Создание задач по итогам созвона

## PURPOSE

Этот скилл нужен, когда после командного созвона требуется:

- прочитать транскрипцию разговора;
- понять, какие решения, проблемы и договорённости были обсуждены;
- изучить кодовую базу текущего проекта;
- задать пользователю уточняющие вопросы;
- предложить рекомендованное Парето-оптимальное решение;
- после ответов пользователя подготовить понятное ТЗ и список задач там, где укажет пользователь.

Главная цель — превратить разговор в практичный план работ, написанный простым языком, как для джуна.

## WHEN TO USE

Используй этот скилл, если пользователь:

- прислал транскрипцию созвона или указал файл с ней;
- просит разобрать итоги созвона;
- просит составить задачи, ТЗ, план реализации или backlog по результатам обсуждения;
- хочет, чтобы ты сначала изучил кодовую базу проекта, а потом оформил задачи.

## DO NOT USE WHEN

Не используй этот скилл, если:

- нет транскрипции и нет ссылки на файл с ней;
- пользователь не хочет задач или ТЗ;
- нужно просто кратко пересказать созвон без анализа проекта;
- пользователь просит игнорировать кодовую базу;
- пользователь уже дал готовое ТЗ, и нужно только отредактировать текст.

## INPUTS

Перед началом работы нужны:

- путь к файлу с транскрипцией;
- путь к кодовой базе или доступ к проекту;
- место, куда нужно записать результат;
- при возможности: название проекта, стек, ветка, окружение, ограничения.

Если чего-то не хватает — задай короткий уточняющий вопрос.

## WORKFLOW

Работай в таком порядке:

### 1. Прочитай транскрипцию

Извлеки из неё:

- цели созвона;
- проблемы;
- решения;
- спорные моменты;
- зависимости;
- риски;
- упомянутые файлы, модули, сервисы, эндпоинты, таблицы, очереди, интеграции.

### 2. Изучи кодовую базу

Нужно не просто быстро посмотреть файлы, а тщательно понять контекст:

- где находится связанный функционал;
- как сейчас устроен поток данных;
- какие есть ограничения;
- какие уже существуют решения;
- где можно внести изменения с минимальным риском;
- что может сломаться при изменении.

Если в проекте много мест, начинай с наиболее вероятных точек изменения, затем расширяй обзор.

### 3. Сопоставь транскрипцию и код

Определи:

- что именно уже есть в проекте;
- чего не хватает;
- какие пункты из созвона реально внедрить быстро;
- какие требуют более крупной переработки;
- где есть технические риски или скрытая сложность.

### 4. Предложи Парето-оптимальное решение

Перед уточняющими вопросами обязательно покажи рекомендованный вариант.

Парето-оптимальное решение — это решение, которое:

- даёт максимальную пользу при минимальной сложности;
- закрывает большую часть ценности задачи;
- не требует лишней архитектурной перестройки;
- может быть реализовано быстрее и безопаснее, чем “идеальный” вариант.

Оформи его так:

- что предлагается сделать;
- почему это лучший баланс между ценой и пользой;
- какие ограничения у такого подхода;
- что останется на следующий этап.

### 5. Задай уточняющие вопросы

После анализа задай только те вопросы, без которых нельзя качественно составить задачи.

Требования к вопросам:

- задавай только действительно важные вопросы;
- не больше 3–7 вопросов за один раз, если возможно;
- вопросы должны быть конкретными;
- рядом с каждым вопросом укажи свой рекомендуемый ответ, если он очевиден.

Формат вопросов:

- вопрос;
- зачем он нужен;
- рекомендуемый вариант, если он очевиден.

### 6. После ответов пользователя составь ТЗ и задачи

Когда пользователь ответит, подготовь результат в понятном виде и запиши его туда, куда он указал.

## OUTPUT REQUIREMENTS

Пиши простым и понятным языком, как для джуна.

Избегай:

- лишней теории;
- сложных слов без необходимости;
- перегруженных формулировок;
- абстрактных фраз вроде “улучшить систему” без расшифровки.

ТЗ должно включать:

- цель;
- контекст;
- текущую проблему;
- предложенное решение;
- список задач;
- порядок выполнения;
- критерии готовности;
- риски и ограничения;
- что не входит в объём работ, если это важно.

Если уместно, разбивай задачи по смыслу:

- подготовка;
- backend;
- frontend;
- интеграции;
- тестирование;
- документация;
- выкладка.

## TASK WRITING RULES

Каждая задача должна быть:

- конкретной;
- проверяемой;
- выполнимой;
- без двусмысленности.

Формат задачи:

- название;
- цель;
- что сделать;
- где делать;
- результат;
- критерий готовности;
- зависимости.

Пример хорошей формулировки:
“Добавить проверку статуса пользователя в middleware, чтобы закрыть доступ к разделу X для неавторизованных пользователей.”

Плохая формулировка:
“Сделать безопасность лучше.”

## QUESTIONING RULES

Если данных не хватает, не выдумывай детали.

Сначала:

- покажи, что ты понял из созвона;
- покажи рекомендованное Парето-решение;
- затем задай вопросы.

Если есть несколько возможных трактовок, явно перечисли их и предложи наиболее вероятную.

## CODEBASE ANALYSIS RULES

При изучении кода:

- ищи реальные точки интеграции;
- опирайся на существующую архитектуру;
- не предлагай решение, которое ломает текущий стиль проекта без необходимости;
- не усложняй, если проблема решается минимальным изменением;
- отмечай места, где нужны тесты и проверка побочных эффектов.

Если есть несколько путей реализации, выбери тот, который:

- проще внедрить;
- легче поддерживать;
- меньше рискует сломать текущий функционал;
- даёт достаточный эффект.

## OUTPUT FORMAT

Всегда используй такой порядок ответа:

1. Краткое понимание созвона.
2. Рекомендованное Парето-оптимальное решение.
3. Что в кодовой базе уже есть и что нужно изменить.
4. Уточняющие вопросы.
5. После ответов — готовое ТЗ и список задач.

Если пользователь просит сразу финальный результат и данных достаточно, пропускай вопросы и сразу составляй ТЗ.

## QUALITY BAR

Проверь, что:

- все выводы опираются на транскрипцию и код;
- нет выдуманных деталей;
- задачи можно передать в работу;
- текст понятен человеку без глубокого погружения в проект;
- рекомендации практичны и не перегружены.

## FAILURE HANDLING

Если транскрипция неполная, кодовая база недоступна или данных недостаточно:

- честно скажи, чего не хватает;
- перечисли, что удалось понять;
- задай минимальный набор уточняющих вопросов;
- не начинай писать ТЗ вслепую.

Если в транскрипции есть противоречия:

- укажи их;
- предложи наиболее вероятную интерпретацию;
- попроси подтверждение.

## EXAMPLES

Хороший результат:

- Понять, что команда обсуждала ускорение регистрации.
- Проверить, как сейчас устроена регистрация в проекте.
- Предложить Парето-решение: не переписывать весь флоу, а добавить недостающую валидацию и один сервисный слой.
- Спросить, нужно ли поддерживать старые сценарии.
- После ответа оформить ТЗ простым языком с задачами по backend, тестам и документации.

Плохой результат:

- Пересказать созвон общими словами.
- Предложить сразу большой рефакторинг без анализа.
- Написать ТЗ с непонятными терминами и без критериев готовности.
