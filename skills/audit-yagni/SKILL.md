## Правило применимости (Relevance Rule)

Применим к любому production-коду с классами, паттернами проектирования или абстракциями. Для простых скриптов-утилит без архитектуры — применяй только к явным over-engineering паттернам.

## Чеклист

| Check ID | Проверка |
|----------|----------|
| YAGNI-01 | Нет закомментированного кода |
| YAGNI-02 | Нет неиспользуемых экспортов |
| YAGNI-03 | Нет абстракций с единственной реализацией без тестов |
| YAGNI-04 | Feature flags не застряли в одном значении |
| YAGNI-05 | Нет TODO/FIXME старше 6 месяцев |

## Правила верификации

1. **Только чеклист**: оценивай ТОЛЬКО проверки выше. Не добавляй новые.
2. **Нет доказательства = ✅ PASS**: не можешь указать `файл:строка` — ставь PASS.
3. **Baseline приоритетен**: check_id есть в `docs/audit-baseline.yml` → `⏸ ACCEPTED`.
4. **Только 🔴/🟠 FAIL требуют решения**: 🟡/🟢 — решение необязательно.

## Baseline

До анализа:
```bash
if [ ! -f ./docs/audit-baseline.yml ]; then
  mkdir -p ./docs
  cp ./skills/audit/audit-baseline-template.yml ./docs/audit-baseline.yml 2>/dev/null || \
    printf "accepted: []\n" > ./docs/audit-baseline.yml
fi
cat ./docs/audit-baseline.yml
```

## Контекст анализа

**Преждевременные абстракции:**
- Интерфейс/абстрактный класс с единственной реализацией (без тестов, где mock нужен)
- Factory/Builder/Strategy для объекта с 1-2 вариантами создания
- Generic-типы с одним конкретным использованием
- Event bus/pub-sub для прямых вызовов между 2 модулями

**Мёртвый код:**
- Экспортируемые функции/классы без единого импорта
- Закомментированный код
- Feature flags, всегда равные true/false
- TODO/FIXME старше 6 месяцев без прогресса

**Излишняя гибкость:**
- Конфигурируемость параметров, которые никогда не меняются
- Plugin-архитектура для системы без плагинов
- Versioning API для внутренних сервисов
- Опциональные параметры, всегда передаваемые с одним значением

**Cargo-cult паттерны:**
- Repository pattern поверх ORM без необходимости абстракции
- Service layer, просто проксирующий вызовы без логики
- DTO для объектов, идентичных entity

## Формат вывода

| Check ID | Проверка | Статус | Доказательство | Решение |
|----------|----------|--------|----------------|---------|
| YAGNI-01 | Нет закомментированного кода | ✅ PASS | — | — |
| YAGNI-02 | Нет неиспользуемых экспортов | ❌ FAIL 🟡 | `lib/formatters.ts:89` | **1. Удалить неиспользуемые экспорты** \\ 2. Добавить ts-prune в CI для автоматического обнаружения \\ 3. Пометить @deprecated и удалить в следующем релизе |
| YAGNI-05 | Нет TODO/FIXME старше 6 месяцев | ⏸ ACCEPTED | `src/auth.ts:34` | В baseline: known tech debt, трекается в Jira PROJ-123 |

Статусы: `✅ PASS` / `❌ FAIL 🔴` / `❌ FAIL 🟠` / `❌ FAIL 🟡` / `❌ FAIL 🟢` / `⏸ ACCEPTED`

Для `❌ FAIL`: ровно 3 варианта решения, разделитель `\\`, вариант 1 жирным.

Если все PASS — выведи: `✅ Over-engineering не обнаружен.`
