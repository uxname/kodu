## Правило применимости (Relevance Rule)

Применим к коду, принимающему внешние данные: HTTP handlers, WebSocket, CLI args, file parsers, event consumers, gRPC endpoints. Для чисто внутреннего кода без внешних входов — верни пустой ответ.

## Чеклист

| Check ID | Проверка |
|----------|----------|
| VAL-01 | Все req.body проходят schema validation (zod/joi/etc) |
| VAL-02 | Все path params типизированы и проверены |
| VAL-03 | Все query params валидированы |
| VAL-04 | Строки имеют maxLength ограничения |
| VAL-05 | Числовые params проверены на диапазон (min/max) |
| VAL-06 | Enum params проверяются по whitelist |
| VAL-07 | JSON.parse в try/catch с валидацией структуры |
| VAL-08 | User ID из аутентифицированного контекста, не из body |

## Правила верификации

1. **Только чеклист**: оценивай ТОЛЬКО проверки выше. Не добавляй новые.
2. **Нет доказательства = ✅ PASS**: не можешь указать `файл:строка` — ставь PASS.
3. **Baseline приоритетен**: check_id есть в `docs/audit-baseline.yml` → `⏸ ACCEPTED`.
4. **Только 🔴/🟠 FAIL требуют решения**: 🟡/🟢 — решение необязательно.

## Контекст анализа

**Отсутствие валидации:**
- HTTP request body используется напрямую без schema validation
- Query params / path params используются без типизации и проверки
- Отсутствие проверки обязательных полей
- Нет валидации типов (строка может прийти вместо числа)

**Недостаточная валидация:**
- Отсутствие maxLength для строк (DoS через огромную строку)
- Нет проверки диапазонов чисел (отрицательные ID, огромные offset)
- Отсутствие whitelist для enum-полей (принимается любое строковое значение)

**Trust boundary нарушения:**
- JWT claims используются без верификации подписи
- User ID берётся из тела запроса вместо аутентифицированного контекста
- Данные из БД возвращаются клиенту без фильтрации чувствительных полей

**Санитизация структуры:**
- JSON.parse без try/catch и валидации структуры
- Данные из БД возвращаются клиенту без фильтрации чувствительных полей (массовое присваивание)

## Граница с другими аудитами

- **HTML/SQL escaping, XSS, path traversal** — зона `audit-owasp`, здесь не дублируй
- **Прямые логические ошибки в валидации** — зона `audit-bugs`
- **Форматы ответов API** — зона `audit-api-contracts`

## Формат вывода

| Check ID | Проверка | Статус | Доказательство | Решение |
|----------|----------|--------|----------------|---------|
| VAL-01 | Все req.body проходят schema validation | ✅ PASS | — | — |
| VAL-04 | Строки имеют maxLength ограничения | ❌ FAIL 🟠 | `handlers/user.ts:22` | **1. Добавить maxLength в zod-схему** \\ 2. Ручная проверка длины в handler \\ 3. Ограничение на уровне БД |
| VAL-08 | User ID из аутентифицированного контекста | ⏸ ACCEPTED | `routes/order.ts:9` | В baseline: legacy endpoint, запланирован рефактор |

Статусы: `✅ PASS` / `❌ FAIL 🔴` / `❌ FAIL 🟠` / `❌ FAIL 🟡` / `❌ FAIL 🟢` / `⏸ ACCEPTED`

Для `❌ FAIL`: ровно 3 варианта решения, разделитель `\\`, вариант 1 жирным.

Если все PASS — выведи: `✅ Валидация граничных данных реализована корректно.`
